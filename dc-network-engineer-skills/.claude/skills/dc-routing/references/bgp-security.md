# BGP Security — RPKI, Prefix Filtering, Bogon Protection

## 1. RPKI (Resource Public Key Infrastructure)

### What Is RPKI?
RPKI cryptographically verifies that a BGP announcement is authorized by the prefix owner. It prevents **route hijacking** — where someone announces your IP space as their own.

### How It Works
```
1. Prefix owner creates a ROA (Route Origin Authorization):
   "AS 65001 is authorized to announce 203.0.113.0/24"

2. ROA is published in an RPKI repository (RIPE, APNIC, ARIN...)

3. Your router runs an RPKI validator (rpki-client, Routinator, FORT)

4. Validator feeds validation results to the router via RTR protocol

5. Router marks each received prefix as:
   - VALID: ROA exists, AS matches → Accept ✅
   - INVALID: ROA exists, AS does NOT match → Reject ❌
   - NOT-FOUND: No ROA exists → Accept with lower preference ⚠️
```

### Juniper RPKI Configuration
```junos
# Configure RPKI validator connection
set routing-options validation group RPKI-VALIDATOR session 10.254.0.100 port 8323
set routing-options validation group RPKI-VALIDATOR session 10.254.0.100 local-address 10.0.0.1

# Apply validation in BGP import policy
set policy-options policy-statement REJECT-INVALID term INVALID from validation-database invalid
set policy-options policy-statement REJECT-INVALID term INVALID then reject
set policy-options policy-statement REJECT-INVALID term DEFAULT then accept

# Apply to BGP group
set protocols bgp group ISP-TRANSIT import [ REJECT-INVALID IMPORT-FROM-ISP ]
```

### Verification
```junos
show validation session                      # RPKI session status
show validation database                     # Cached ROAs
show validation statistics                   # Counters
show route protocol bgp validation-state invalid  # Routes that would be rejected
```

## 2. Prefix Filtering (IRR-Based)

### What Is IRR?
The Internet Routing Registry (IRR) is a database where network operators register their routing policy (which prefixes they announce, which ASes are their customers). Tools like `bgpq4` can generate prefix-lists from IRR data.

### Generate Prefix Filters from IRR
```bash
# Install bgpq4
# apt install bgpq4  (or brew install bgpq4)

# Generate a Juniper prefix-list for a customer's AS-SET
bgpq4 -Jl CUSTOMER-PREFIXES AS-EXAMPLE
# Output:
# policy-options {
#     prefix-list CUSTOMER-PREFIXES {
#         203.0.113.0/24;
#         198.51.100.0/23;
#     }
# }

# Generate for a specific ASN
bgpq4 -Jl ISP-PREFIXES AS7552
```

### Apply Prefix Filters on Juniper
```junos
# Define prefix-list (generated by bgpq4 or manually)
set policy-options prefix-list ISP-A-ALLOWED 203.0.113.0/24
set policy-options prefix-list ISP-A-ALLOWED 198.51.100.0/23

# Import policy: only accept prefixes in the list
set policy-options policy-statement IMPORT-FROM-ISP-A term ALLOWED from prefix-list ISP-A-ALLOWED
set policy-options policy-statement IMPORT-FROM-ISP-A term ALLOWED then accept
set policy-options policy-statement IMPORT-FROM-ISP-A term REJECT-REST then reject
```

## 3. Bogon Filtering

### What Are Bogons?
Bogon prefixes are IP addresses that should **never appear in the public internet routing table**. They include private addresses (RFC 1918), reserved ranges, and unallocated space.

### Bogon Prefix List
```junos
set policy-options prefix-list BOGONS 0.0.0.0/8 orlonger          # "This" network
set policy-options prefix-list BOGONS 10.0.0.0/8 orlonger         # RFC 1918 private
set policy-options prefix-list BOGONS 100.64.0.0/10 orlonger      # Shared Address Space (CGN)
set policy-options prefix-list BOGONS 127.0.0.0/8 orlonger        # Loopback
set policy-options prefix-list BOGONS 169.254.0.0/16 orlonger     # Link-local
set policy-options prefix-list BOGONS 172.16.0.0/12 orlonger      # RFC 1918 private
set policy-options prefix-list BOGONS 192.0.0.0/24 orlonger       # IETF Protocol Assignments
set policy-options prefix-list BOGONS 192.0.2.0/24 orlonger       # Documentation (TEST-NET-1)
set policy-options prefix-list BOGONS 192.168.0.0/16 orlonger     # RFC 1918 private
set policy-options prefix-list BOGONS 198.18.0.0/15 orlonger      # Benchmarking
set policy-options prefix-list BOGONS 198.51.100.0/24 orlonger    # Documentation (TEST-NET-2)
set policy-options prefix-list BOGONS 203.0.113.0/24 orlonger     # Documentation (TEST-NET-3)
set policy-options prefix-list BOGONS 224.0.0.0/4 orlonger        # Multicast
set policy-options prefix-list BOGONS 240.0.0.0/4 orlonger        # Reserved (Class E)

# Reject bogons in import
set policy-options policy-statement REJECT-BOGONS term BOGONS from prefix-list BOGONS
set policy-options policy-statement REJECT-BOGONS term BOGONS then reject
```

### Bogon AS Numbers
```junos
# Reject routes with private or reserved AS numbers in the path
set policy-options as-path BOGON-AS ".*(0|23456|[64496-131071]|[4200000000-4294967295]).*"
set policy-options policy-statement REJECT-BOGON-AS term BAD-AS from as-path BOGON-AS
set policy-options policy-statement REJECT-BOGON-AS term BAD-AS then reject
```

## 4. Max-Prefix Protection

Limits the number of prefixes accepted from a BGP peer. Prevents a misconfigured peer from sending the full internet table (900K+ prefixes) to a device that can't handle it.

```junos
# Tear down session if peer sends > 10000 prefixes
set protocols bgp group ISP-TRANSIT neighbor 203.0.113.1 family inet unicast prefix-limit maximum 10000
set protocols bgp group ISP-TRANSIT neighbor 203.0.113.1 family inet unicast prefix-limit teardown 90  # Warn at 90%

# For a customer that should only send 100 prefixes
set protocols bgp group CUSTOMER neighbor 198.51.100.1 family inet unicast prefix-limit maximum 100
set protocols bgp group CUSTOMER neighbor 198.51.100.1 family inet unicast prefix-limit teardown 80
```

## 5. Additional BGP Security Practices

### AS-PATH Filtering
```junos
# Only accept routes originated by the peer's AS (for single-homed customers)
set policy-options as-path CUSTOMER-ONLY "^65002$"
set policy-options policy-statement IMPORT-FROM-CUSTOMER term AS-CHECK from as-path CUSTOMER-ONLY
set policy-options policy-statement IMPORT-FROM-CUSTOMER term AS-CHECK then accept
set policy-options policy-statement IMPORT-FROM-CUSTOMER term REJECT then reject
```

### BGP Authentication (TCP MD5)
```junos
set protocols bgp group ISP-TRANSIT neighbor 203.0.113.1 authentication-key "SecretKey123"
```

### GTSM (Generalized TTL Security Mechanism)
Protects against BGP session spoofing by requiring TTL=255 (must be directly connected):
```junos
set protocols bgp group ISP-TRANSIT neighbor 203.0.113.1 ttl 255
set protocols bgp group ISP-TRANSIT neighbor 203.0.113.1 multihop no  # eBGP default
```

### Minimum Prefix Length
```junos
# Don't accept prefixes more specific than /24 (internet standard)
set policy-options policy-statement REJECT-TOO-SPECIFIC term TOO-SPECIFIC from route-filter 0.0.0.0/0 prefix-length-range /25-/32
set policy-options policy-statement REJECT-TOO-SPECIFIC term TOO-SPECIFIC then reject
```

## 6. Comprehensive BGP Import Policy (Template)

Combine all protections into a single import policy chain:

```junos
# Apply in order: bogons → bogon-AS → RPKI → max-prefix → prefix-filter → community-tag → accept
set protocols bgp group ISP-TRANSIT import [ REJECT-BOGONS REJECT-BOGON-AS REJECT-INVALID REJECT-TOO-SPECIFIC IMPORT-FROM-ISP ]
```

> **Best practice**: Apply security filters in this order: Bogons first (cheapest check), then RPKI (crypto check), then IRR prefix-list (most specific). This minimizes processing for obviously bad routes.
